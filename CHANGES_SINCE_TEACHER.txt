§0  起点：老师 L4 分支 19a26cf 提了什么
================================================================

老师在自己的 L4 分支上重写了 and.cpp，核心是两件事：

  (a) 抽出 SideCtx 结构 + make_side / advance_side 工厂
      —— 把 AND 二元算子里每个"边"（A 和 B）要用的所有指针
      / fill_vec / cursor 打包成一个 struct，让两边的代码读起来
      完全对称。

  (b) 加 batch L4 bypass
      —— 一次同时检测 A 边和 B 边的 L4 batch（8 字节，覆盖 64
      个 region = 4096 word）。只要任意一边在这 64 个 region
      内全部 0，AND 结果就是 0：跳过这一整批的 L3->L2->L1
      cascade，只推进另一边的 cursor，然后写 4096 字节的 0。

我们把这份老师的 and.cpp 整段复制过来作为基线，再做下面的改动。


================================================================
§1  老师当面口头要求的 3 件事（针对 and.cpp）
================================================================

老师在 D8F04A2B 这张图里圈出来三处，要求改：

────────────────────────────────────────────────────────────────
① "SideCtx A = make_side(*this, a_l1)" → "this->make_side(a_l1)"
────────────────────────────────────────────────────────────────
  老师原话：
    "图中的 SideCtx A = make_side(*this, a_l1); 或者 B 这种，是
     方便我理解的，实际上我改完后就可以直接用类似 this 的这种"

  含义：
    工厂函数应该作为成员函数，而不是独立函数靠 *this 传进去。
    A/B 这个别名可以保留（让 A、B 两条路径视觉对称），但是
    factory 的调用要写成 this->make_side(...) / other.make_side(...)。

  对应改动位置：
    [src/core/combit/src/and.cpp:72-73]
      SideCtx A = this->make_side(a_l1);
      SideCtx B = other.make_side(b_l1);

────────────────────────────────────────────────────────────────
② SideCtx + make_side + advance_side 提到 combit.h，因为共用
────────────────────────────────────────────────────────────────
  老师原话：
    "图中这一段最好提到 combit.h，提到外面，因为是共用的函数"

  含义：
    老师只在 and.cpp 里写了 SideCtx 和两个工厂，但 or.cpp / xor.cpp
    都要用同样的结构。所以应该把它们提到头文件作为类的内部 helper，
    三个 .cpp 共享一份定义。

  对应改动位置：
    [src/core/combit/include/combit.h:107-121]   struct SideCtx
    [src/core/combit/include/combit.h:131-147]   inline SideCtx make_side(...)
    [src/core/combit/include/combit.h:153-160]   static inline void advance_side(...)

  注意：advance_side 是 static inline 的，因为它只读 L2/L1 buffer
  和 cursor，不访问 this 的私有字段。

────────────────────────────────────────────────────────────────
③ 删掉 "Sync the SideCtx cursors back" 那一段
────────────────────────────────────────────────────────────────
  老师原话：
    "然后老师说就像图二（D8F04A2B...）的框住的这一部分就可以删掉了"

  含义：
    老师原版 and.cpp 在 SIMD 主循环结束后，有一段把 SideCtx 里
    的 cursor 复制回外层 a_l3_lit_off / b_l3_lit_off 等局部变量，
    给 scalar tail 用。现在 scalar tail 也直接读 SideCtx 字段
    （A.l3_lit_off 等），这段同步就多余了。

  对应改动位置：
    [src/core/combit/src/and.cpp:232-241]
      scalar tail 直接通过 A.l3_fill_ones / A.l3_lits / A.l3_lit_off
      等访问，不再走"先 sync 出来再用"的中间变量。
      原来那段同步代码已经物理删除。

────────────────────────────────────────────────────────────────
附带（同一次重构）：和老师对齐的命名
────────────────────────────────────────────────────────────────
  老师在自己的 and.cpp 里把字段名都做了简化：
      l1_literals_       → l1_lits_
      l1_literal_count_  → l1_lit_count_
      l1_literal_data()  → l1_lit_data()
      num_literals()     → num_lits()
      （L2/L3 同理）

  老师原话：
    "完全按照老师的 and 里面的命名方式，来改 combit.h 的命名"

  对应改动位置：
    [src/core/combit/include/combit.h:94-98]  访问器名字
    [src/core/combit/include/combit.h:186-197] 私有字段名字
    [src/core/combit/src/combit.cpp 全文]      所有引用同步改名
    [src/core/combit/src/and.cpp / or.cpp / xor.cpp 全文] 同步

§2  把同一模式扩展到 or.cpp / xor.cpp
================================================================

老师原话：
  "然后老师说，按照 and 的方式，把 or 和 xor 也改成这种"

含义：
  or.cpp 和 xor.cpp 原本各自维护一套 a_l3_lits / a_l2_lits_off /
  ... 散落的局部变量，可读性差。改成统一调用 combit.h 的 SideCtx
  + make_side + advance_side。

────────────────────────────────────────────────────────────────
or.cpp 的对齐
────────────────────────────────────────────────────────────────
  [src/core/combit/src/or.cpp:56-59]
    SideCtx A = this->make_side(a_l1);
    SideCtx B = other.make_side(b_l1);

  per-region "OR：0 | x = x" bypass 也用 advance_side 跳 cursor：
    [src/core/combit/src/or.cpp:129-181]
      a 端 zero → 只 expand B
      b 端 zero → 只 expand A
      两边都 zero → 写零

  in-place operator|=（[src/core/combit/src/or.cpp:437-439]）只对
  RHS 用 SideCtx（LHS 是 Decompressed，本身就是一片连续的 L1）。

────────────────────────────────────────────────────────────────
xor.cpp 的对齐
────────────────────────────────────────────────────────────────
  [src/core/combit/src/xor.cpp:54-67]  SideCtx A / B + 同款 bypass 常量
  [src/core/combit/src/xor.cpp:79-130] per-region "XOR：0 ^ x = x" bypass

  XOR 不做 "1 ^ x = ~x" 的 bypass（"取反"不比 "做 xor" 快多少，
  额外分支不划算）—— 这与老师 "L2/L1 基本上没用了" 的思路一致。

§3  今天 (2026-05-15) 额外补的 3 处 bypass
================================================================

我自己审计了一遍 bypass 真值表，发现两个改进空间，跟老师的方向
一致（都是 L3/L4 驱动，不读 L1/L2 字节），加进去。

────────────────────────────────────────────────────────────────
③.1  or.cpp 加 batch L4 bypass（镜像老师 AND 的 batch bypass）
────────────────────────────────────────────────────────────────
  动机：
    AND 在 [src/core/combit/src/and.cpp:124-156] 已有 batch 级
    的"任一边 L4 全 0 → 整批 4096 word 写 0"快路径。
    OR 之前没有对应物——每个 region 都要进 per-region 循环判一次
    l3a == 0。在 sparse OR 里这浪费 ~64 个 cmp 每 batch。

  新增逻辑（只处理对称的 "both zero" 情况，简单干净）：
    [src/core/combit/src/or.cpp:96-110]
      if (a_struct_zero && b_struct_zero
          && a_l4_mask == 0 && b_l4_mask == 0) {
          memset(r_l1 + r_off, 0, batch_size * 64);
          r_off += batch_size * 64;
          continue;
      }

  作用：
    两边都 structurally zero（l1/l2/l3 fill 全 0）且 batch 的
    L4 mask 全 0 → 这 4096 word 一定全是 0|0=0 → 一次 memset 解决。
    Cursor 不动（popcount(0)=0，l3 全 0 也不消费 L2/L1）。

  收益预期：
    c=2000 这种 sparse 场景，理论 batch 命中率 ~27%，能省 64×
    per-region 检查 + 64 次 64B store。受 1-iter 噪声覆盖，
    定量得跑 iter≥20。

────────────────────────────────────────────────────────────────
③.2  or.cpp 加 per-region "1 wins" bypass（OR 真值表第 4 种）
────────────────────────────────────────────────────────────────
  动机：
    OR 真值表 4 种情况：A=0→B、B=0→A 已经做了；A=1→1、B=1→1
    一直没做，原来一直走完整 expand+OR。

  关键观察（这就是为什么老师说"L1/L2 不用 bypass"也不冲突）：
    "Region 隐式全 1" 的判定只用三个东西：
      - l3a == 0           （per-region 信息，已 expand 出来）
      - !l2_fill_ones       （side-wide 常量）
      - l1_fill_ones        （side-wide 常量）
    完全不需要去扫 L2/L1 字节流。

  新增逻辑：
    [src/core/combit/src/or.cpp:72-73]   常量计算
      const bool a_ones_when_l3_zero = !A.l2_fill_ones && A.l1_fill_ones;
      const bool b_ones_when_l3_zero = !B.l2_fill_ones && B.l1_fill_ones;

    [src/core/combit/src/or.cpp:183-215] per-region 新分支
      if (a_ones_when_l3_zero && l3a == 0) {
          advance_side(B, l3b);           // 跳 B 的 cursor
          // 写 64 字节 0xFF
      }
      // B 端对称

  代价：
    我们 sparse-1 benchmark 里 l1_fill_ones=false，
    a_ones_when_l3_zero 是 loop-invariant false → 这两个 if 是
    永远不进的死分支，branch predictor 一次就学会。每 region 多
    ~1 cycle，可忽略。

────────────────────────────────────────────────────────────────
③.3  and.cpp 加 per-region "1 wins" bypass（AND 真值表第 4 种）
────────────────────────────────────────────────────────────────
  对称地，AND 之前只做了"任一边 l3==0 且 zero → 结果 0"，没做
  "任一边 l3==0 且全 1 → 结果 = 另一边"。

  新增逻辑：
    [src/core/combit/src/and.cpp:100-101] 常量计算（与 OR 同款）
    [src/core/combit/src/and.cpp:197-261] per-region 三个新分支
      a_is_ones && b_is_ones → 写 64 字节 0xFF（cursor 都不动）
      a_is_ones → expand B 写入结果（A 端没 L2/L1 要跳）
      b_is_ones → expand A 写入结果

  和 ②.2 同样：sparse-1 benchmark 里全是死分支，性能成本可忽略。

────────────────────────────────────────────────────────────────
正确性回归测试：
────────────────────────────────────────────────────────────────

§4  配套：benchmark / 数据生成 / 出图工具
================================================================

这部分是为了能画出"4 种 PDF 对比图"做的配套，不是 ComBit 核心算法
本身的改动，但答辩时如果被问到"图是怎么画的"，可以这样讲：

────────────────────────────────────────────────────────────────
④.1  benchmark：加 --compress-results 开关
────────────────────────────────────────────────────────────────
  [src/benchmark/benchmark_main.cpp:1130-1133]
    --compress-results 把 ComBit 全局开关 combit_compress_results
    置 true → operator|&^ 返回 Compressed 结果（默认是 Decompressed）。

  [src/benchmark/benchmark_main.cpp:1165-1168]
    根据 flag 给 ComBit 一个区分的标签 "ComBIT (compress)"
    （默认是 "ComBIT (New)"），便于一份 CSV 里同时承载两种模式
    的数据，画图时按标签筛。

────────────────────────────────────────────────────────────────
④.2  数据生成：3 种 stress bitmap
────────────────────────────────────────────────────────────────
  为了让 CRoaring 走出"array → bitset → array regret"和
  "array merge fast path"等不同路径，做了三种合成位图：

  [util/gen_bitmap.cpp:88-125]
    transition 模式 (-t <count_a>)：两个独立 random，每 65536-segment
    各 count_a 个 bit。t3500 / t16000 用这个。

  [util/gen_bitmap.cpp:137 起]   make_asymmetric_disjoint_buckets
  + 新增 -A <count_a> -B <count_b> 命令行：A 有 count_a 个、B 有
    count_b 个、A∩B=∅。A2500_B100 用这个。

  (overlap 模式 -o 之前就有：A 和 B 共享 95% 的 bit + 5% 互不相交
   的 fresh，o2200 用这个。)

  [src/benchmark/uti.cpp:110-132]
    解析器加上 'A' tag，识别 bm_100m_A{a}_B{b}_{algo} 目录名。

────────────────────────────────────────────────────────────────
④.3  出图：plot_4pdf.py
────────────────────────────────────────────────────────────────
  [tools/plot_4pdf.py]  ← 新写的脚本
    读 /tmp/plot_results.csv，按
      operation ∈ {OR_op, AND_op}
      result_mode ∈ {decompress, compress}
      y_scale ∈ {log, linear}
    8 种组合，输出 8 张 PDF。

  3 个 OR engineered 点 (t3500 / A2500_B100 / o2200) 和 1 个 AND
  engineered 点 (t16000) 与标准 sweep 在 *同一条折线* 上，按
  count_a / 65536 折算成密度坐标，marker 与标准 sweep 完全一致
  （不区别画大小），这样 CRoaring 的"down-up-down"形态直接体现
  在线的形状上。


================================================================
答辩时一句话总结
================================================================

  "起点是老师在 L4 分支提的 batch L4 bypass + SideCtx 抽象。
   按老师 3 条要求做了重构：(1) make_side 改成成员函数 this 调用，
   (2) SideCtx/make_side/advance_side 提到 combit.h 公共位置，
   (3) 删除冗余 cursor 同步段。同时把 and.cpp 的命名风格统一
   到整个 combit 模块，并把同一模式扩展到 or.cpp / xor.cpp。
   今天我自己补了 bypass 真值表的剩余 50%：OR 加 batch L4
   两边都零的快速 memset 路径，AND/OR 都加 per-region 1-side
   wins 分支——这些都是 L3/L4 驱动的判定，与老师'L1/L2 不用
   做 bypass'的设计原则不冲突，cost 是 sparse-1 场景下的死分支
   ~1 cycle/region，可忽略。"
